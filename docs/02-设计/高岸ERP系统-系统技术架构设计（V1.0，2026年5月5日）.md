## 1. 总体架构

### 1.1 架构分层

系统采用前后端分离 + 微服务架构，按业务域划分为以下层次：

```
┌─────────────────────────────────────────────────┐
│                  客户端层                         │
│   客人端(小程序)  店员端(小程序)  管理后台(PC/Web) │
└──────────────────────┬──────────────────────────┘
                       │ HTTP/REST + WebSocket
┌──────────────────────▼──────────────────────────┐
│                 API网关层                         │
│            Nginx + 统一认证鉴权                    │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│                 业务服务层                         │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐  │
│  │营销域 │ │消费域 │ │收入域 │ │支出域 │ │会员域 │  │
│  └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘  │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐  │
│  │对账域 │ │巡检域 │ │资产域 │ │人事域 │ │采购域 │  │
│  └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘ └──┬───┘  │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐           │
│  │报表域 │ │多门店 │ │预警AI│ │物联网 │           │
│  └──────┘ └──────┘ └──────┘ └──────┘           │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│                公共服务层                         │
│  工作流引擎  告警中心  消息推送  文件服务  定时任务 │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│                 数据存储层                         │
│   MySQL(主库)  Redis(缓存)  ES(日志/搜索)  对象存储 │
└─────────────────────────────────────────────────┘
```

### 1.2 三层架构说明

| 层级 | 职责 | 技术选型建议 |
|------|------|-------------|
| **客户端层** | 客人端小程序、店员端小程序、管理后台（PC Web） | uni-app（跨端框架）、Vue 3（管理后台） |
| **API网关层** | 统一入口、认证鉴权、限流、路由转发 | Nginx / Spring Cloud Gateway |
| **业务服务层** | 各业务域独立服务，按域拆分 | Spring Boot / Python FastAPI（AI服务） |
| **公共服务层** | 工作流引擎、消息推送、文件存储等通用能力 | Activiti/Flowable、WebSocket、MinIO |
| **数据存储层** | 业务数据、缓存、日志、文件 | MySQL 8.0、Redis 7.x、Elasticsearch、MinIO |

## 2. 业务服务拆分

### 2.1 微服务清单

| 服务名称 | 所属域 | 核心能力 | 数据独立性 |
|---------|-------|---------|-----------|
| `mkt-svc` | 营销域 | 活动管理、优惠券、渠道分析、平台同步 | 独立库 |
| `cust-svc` | 消费域 | 门店展示、预约、点单、支付、退房 | 独立库 |
| `rev-svc` | 收入域 | 收入归集、日结、会计凭证、支付方式配置 | 独立库 |
| `exp-svc` | 支出域 | 请款/报销、供应商、广告费归集、费用分摊 | 独立库 |
| `mem-svc` | 会员域 | 注册、充值、消费、退费、报表 | 独立库 |
| `rec-svc` | 对账域 | 实时对账、差异工单、平台周期对齐 | 独立库 |
| `ins-svc` | 巡检域 | 模板、任务、整改 | 独立库 |
| `ast-svc` | 资产域 | 资产卡片、折旧、盘点 | 独立库 |
| `hr-svc` | 人事域 | 员工、考勤、排班、薪资汇总 | 独立库 |
| `pur-svc` | 采购库存域 | 商品、采购、入库、出库、库存预警、调拨 | 独立库 |
| `rpt-svc` | 报表域 | 月结报告、财务看板、图表 | 独立库 |
| `mul-svc` | 多门店域 | 门店管理、合并报表、股东分红 | 独立库 |
| `iot-svc` | 物联网域 | 设备指令、事件处理、状态监控 | 独立库 |
| `aiw-svc` | 预警AI域 | 异常检测、AI策略建议 | 独立库 |
| `wf-svc` | 工作流引擎 | 审批流程定义、执行、审计 | 共享库 |
| `notify-svc` | 消息推送 | 小程序推送、飞书推送、短信 | — |
| `gateway` | API网关 | 路由、认证、限流、日志 | — |

### 2.2 服务间通信

| 通信方式 | 使用场景 | 技术选型 |
|---------|---------|---------|
| **同步REST** | 实时查询、短事务操作 | OpenFeign + HTTP/2 |
| **异步消息** | 跨域事件通知（订单完成→收入归集、退房→IoT关设备） | RabbitMQ / RocketMQ |
| **事件驱动** | 审计日志、告警通知 | 消息队列 + 事件总线 |
| **定时任务** | 日结、月结、对账、广告费拉取 | XXL-Job / Quartz |

### 2.3 关键事件流

```
订单支付成功 → cust-svc → 发消息 [OrderPaidEvent]
  ├─ rev-svc 订阅 → 生成收入流水
  ├─ iot-svc 订阅 → 下发门禁密码
  └─ notify-svc 订阅 → 推送开门通知给客人

客人退房 → cust-svc → 发消息 [OrderCheckedOutEvent]
  ├─ rev-svc 订阅 → 结算/多退少补
  ├─ iot-svc 订阅 → 关闭房间设备
  ├─ pur-svc 订阅 → 扣减商品库存
  └─ ins-svc 订阅 → 生成保洁任务

每日凌晨00:05 → 定时任务 → rev-svc
  └─ 日结汇总 → 生成会计凭证 → 对账

每月25日01:00 → 定时任务 → rpt-svc
  └─ 检查差异工单 → 月结计算 → 生成报告 → 推送
```

## 3. 数据架构

### 3.1 数据库策略

| 策略 | 说明 |
|------|------|
| **数据库** | MySQL 8.0，每个微服务独立数据库（Database per Service） |
| **缓存** | Redis 7.x，缓存热点数据（门店列表、商品信息、会员会话） |
| **搜索引擎** | Elasticsearch，存储订单检索、日志分析、审计日志 |
| **文件存储** | MinIO（私有化部署）或阿里云OSS，存储凭证图片、巡检照片、报表文件 |
| **消息队列** | RabbitMQ，跨服务异步通信 |
| **搜索引擎** | ES用于订单/日志检索，财务报表从MySQL直接查询 |

### 3.2 多租户（门店隔离）

```
数据隔离方式：共享数据库 + store_id 字段隔离
原因：门店数量有限（当前3家，未来预计10-20家），独立库运维成本过高
缓存隔离：Redis key 前缀为 store:{store_id}
```

### 3.3 结算周期处理

```
数据库设计原则：
- 所有订单、收入流水、支出记录均包含 结算周期开始日期 和 结算周期结束日期 字段
- 月结时根据 结算周期 字段筛选，而非 创建时间
- 跨周期订单通过 周期标记 字段标识归属
```

## 4. 物联网架构

### 4.1 三层层级

| 层级 | 实现 | 说明 |
|------|------|------|
| **业务控制层** | iot-svc（业务服务） | 根据业务事件生成控制指令，接收设备事件 |
| **设备管理层** | 智能家居网关 | 指令转换、设备绑定、离线规则引擎 |
| **物理设备层** | 门锁/空调/灯光/窗帘/音响/传感器 | 执行物理操作 |

### 4.2 通信方式

```
iot-svc ←HTTP/REST→ 智能家居网关 ←Zigbee/WiFi/BT→ 物理设备
```

- ERP与网关之间通过局域网REST API通信（需内网安全策略）
- 网关与设备之间根据设备类型选择Zigbee（低功耗传感器）/ WiFi（门锁、空调）/ 蓝牙（音响）
- 离线自动化规则预配置在网关本地，断网时独立执行

## 5. 工作流引擎

### 5.1 整合方式

| 组件 | 技术选型 | 用途 |
|------|---------|------|
| **工作流引擎** | Flowable（开源，社区活跃） | 审批流程定义、执行、审计 |
| **流程定义** | BPMN 2.0 XML | 请款/报销审批、资产变动、月结审批 |
| **流程存储** | Flowable内建表（ACT_* 系列） | 独立schema部署 |

### 5.2 审批流程类型

| 流程 | 节点数 | 条件分支 | 会签支持 |
|------|-------|---------|---------|
| 支出审批（请款/报销） | 2-4 | 金额阈值路由 | 是 |
| 资产变动审批 | 2-3 | 资产类型 | 是 |
| 会员退费审批 | 2 | 金额阈值 | 否 |
| 月结审批 | 2 | 差异工单状态 | 是 |

## 6. 客户端架构

### 6.1 多端实现方案

| 端 | 技术栈 | 说明 |
|----|--------|------|
| 客人端 | uni-app（微信小程序） | 一套代码多端发行，当前目标微信小程序 |
| 店员端 | uni-app（微信小程序） | 功能较客人端复杂，含工作台、房态管理、对账工单 |
| 店长端 | uni-app（微信小程序） | 报表查看、审批、员工管理 |
| 投资人端 | uni-app（微信小程序） | 简化视图，财务看板、分红查看 |
| 总部/管理后台 | Vue 3 + Element Plus | PC端完整管理功能 |

### 6.2 API调用规范

```
所有客户端通过 API网关 （/api/v1/{domain}/...）调用后端服务
认证方式：微信登录态 + JWT Token
文件上传：直接上传至MinIO/OSS，CLI端获取URL后提交到业务服务
数据同步：关键数据（订单、工单）支持离线缓存，网络恢复后同步
```

## 8. 安全设计

| 关注点 | 措施 |
|--------|------|
| **认证** | 微信登录态 + JWT Token（短期有效） |
| **授权** | RBAC（角色-权限-资源），细粒度到按钮/API级别 |
| **数据隔离** | 门店级数据通过store_id过滤，投资人级数据脱敏 |
| **敏感数据** | 门禁密码传输/存储加密，手机号中间4位脱敏 |
| **审计日志** | 所有写操作记录操作人、时间、IP、内容，保留≥1年 |
| **API安全** | 网关层统一限流、防重放、SQL注入过滤 |

## 9. 非功能设计

### 9.1 性能目标

| 指标 | 目标值 | 实现策略 |
|------|-------|---------|
| 小程序页面响应 | ≤2秒 | CDN静态资源 + API响应缓存 + 数据预加载 |
| IoT指令延迟 | ≤3秒 | 局域网REST调用，避免公网中转 |
| 对账拉取频率 | 每5分钟 | 定时任务队列 |
| 月结报告生成 | ≤5分钟 | 异步处理 + 并行计算 |

### 9.2 可靠性

| 要求 | 措施 |
|------|------|
| 门锁离线可用 | 离线动态密码（时间同步算法） |
| 月结失败告警 | 失败重试3次 + 飞书/小程序告警 + 手动触发 |
| 数据库容灾 | 每日自动备份 + 保留30天 + 主从切换 |
| 服务降级 | 网关限流 + 熔断（Sentinel/Hystrix） |
