# 高岸ERP系统需求说明书 V10.0 — 框架计划

## 总体思路

从"经营好一个连锁茶室体系"的角度出发，**先定大局、再深入细节**。按标准企业部门设置划分业务域，每个域下的功能需求写深写透，确保每一处都可评审、可追溯。

---

## 新章节框架

### 第一章：业务总览
> 定位：回答"这个系统是做什么的、为谁做、核心理念是什么"

| 节 | 内容 | 来源/对比V9.3 |
|----|------|--------------|
| 1.1 项目背景 | 行业痛点、连锁茶室的管理现状、为什么要建ERP | 保留并精简 |
| 1.2 经营模式概览 | 连锁茶室整体价值链：**引流 → 消费 → 结算 → 管理**，四大环节的关系。让读者先看清全貌。 | **新增** |
| 1.3 核心目标 | G1 全自动月结、G2 IoT全流程自动化 | 保留 |
| 1.4 用户角色定义 | 客人、店员、店长、总部运营、财务、投资人，每个角色的职责简述 | 从第二章提取前置 |
| 1.5 设计原则 | **运营驱动**（非财务驱动）、连锁架构、最简可用IoT | 修改 |

> 与V9.3变化：新增"经营模式概览"让人先看全局；删除"标准会计科目""可扩展"等不言自明的原则；修正"财务驱动"为"运营驱动"。

---

### 第二章：业务域与子系统
> 定位：回答"系统覆盖哪些业务范围、怎么划分、一期做什么"

| 节 | 内容 | 来源/对比V9.3 |
|----|------|--------------|
| 2.1 业务域划分 | 按组织部门划分7大业务域：**运营管理域、市场营销域、会员管理域、供应链域、财务域、技术域、人力资源域**。每个域简要说明职责范围。 | **重构**（替换错误的业务域关系图） |
| 2.2 子系统定义 | 对应业务域定义子系统：门店运营系统、营销系统、会员系统、进销存系统、财务系统、IoT平台、人事系统。说明各子系统职责边界。 | **新增**（评审要求先定义子系统再谈边界） |
| 2.3 系统边界与一期范围 | 包含/不包含清单，分期规划 | 保留并优化 |

> 与V9.3变化：删除"前端业务域""核心域""收入域/对账域/报表域"等错误概念；改为按部门划分的7大业务域；新增子系统定义章节回应评审意见。

### 第三章：功能需求（核心章节）
> 定位：按业务域逐项展开功能需求，每个功能点写清**功能描述、前置条件、操作流程、业务规则、异常处理**，确保可评审。

| 节 | 内容 | 来源/对比V9.3 |
|----|------|--------------|
| 3.1 运营管理域 | 包间预约、商品零售、房态管理、保洁任务、门店巡检 | 原第二章客人端+店员端+店长端内容重组 |
| 3.2 市场营销域 | 营销活动管理、优惠券管理、渠道分析 | 新增详细展开 |
| 3.3 会员管理域 | 注册与档案、充值管理、消费与积分、退费管理 | 细化 |
| 3.4 供应链域 | 采购管理、库存管理、供应商管理 | 细化 |
| 3.5 财务域 | 收入管理、支出管理（请款/报销）、自动月结、对账管理、股东分红、报表体系 | 重组细化 |
| 3.6 技术域 | IoT设备管理、智能场景控制、AI预警与策略 | 细化 |
| 3.7 人力资源域 | 考勤管理、薪资核算、员工档案 | 细化 |

每个功能点格式示例：

```
### 3.1.1 包间预约

**功能描述**：客人通过小程序选择门店、时段、包间并完成支付。

**前置条件**：
- 客人已是小程序注册用户

**操作流程**：
1. 选择门店 → 展示可预约时段
2. 选择包间 → 展示价格
3. 确认订单 → 支付
4. 生成预约单 → 推送通知

**业务规则**：
- 预约精度30分钟，可提前2小时至30天
- 超预约时间30分钟未开门自动取消
- 提前2小时可免费取消，否则扣减余额

**异常处理**：
- 支付成功系统故障 → 生成待确认订单人工处理
- 重复预约同一时段 → 系统阻止

**关联流程**：详见4.1客人消费流程
```

> 与V9.3变化：原第二章功能菜单树（按角色分组）非常好，保留其内容，但重新按7大业务域组织，并大幅增加每个功能点的细节深度。这样既保留了框架，又回应了"每个节点都需要评审"的要求。

### 第四章：核心业务流程（泳道图）
> 定位：跨角色、跨系统的业务流程视图，用真正的泳道图呈现。

| 节 | 流程 | 涉及域 |
|----|------|--------|
| 4.1 | 客人预约到店消费流程（含IoT自动化） | 运营+技术+财务 |
| 4.2 | 营销活动创建与发布流程 | 市场+财务 |
| 4.3 | 月度自动结账流程（含对账） | 财务+运营 |
| 4.4 | 请款与报销审批流程 | 财务+各域 |
| 4.5 | 采购与入库流程 | 供应链+财务 |
| 4.6 | 考勤管理流程 | 人力+运营 |
| 4.7 | 门店巡检流程 | 运营 |
| 4.8 | 设备故障处理流程 | 技术+运营 |

每个流程先文字说明再配泳道图。泳道图使用真正的泳道图格式（按角色分列），不再是 `graph LR + subgraph`。

> 与V9.3变化：8个流程保留，但需重新绘制为真正泳道图；4.2营销活动流程需重新审视需求合理性（评审人质疑：美团/抖音是活动发起方，同步方案是否成立）。
